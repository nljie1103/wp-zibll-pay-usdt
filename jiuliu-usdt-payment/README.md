# 九流网络 USDT 支付

这是为当前站点的 WordPress 7.0.1 与 Zibll 9.0 定制开发的独立插件。它不会修改子比主题源码，主题升级不会直接覆盖插件文件。

## 已适配范围

- 子比余额充值
- 年费、终身等会员订单
- 付费文章、下载资源、图片和视频
- 子比商城单品、购物车合并订单、多作者订单与自动发货
- 论坛付费关注
- 子比允许的游客付费订单

插件把 USDT 注册为子比原生现金网关。链上确认后调用 Zibll 9.0 的统一订单完成入口，由子比自己执行余额流水、会员开通、内容授权、销量、返佣、分成、邮件和商城发货。

## 安装

1. WordPress 后台进入“插件 -> 安装插件 -> 上传插件”。
2. 上传 jiuliu-usdt-payment.zip 并启用。
3. 打开左侧“USDT 收款 -> 设置”。
4. 填写 TRON 主网公开收款地址。
5. 填写固定汇率，例如 1 USDT = 7.20 CNY。
6. 建议填写 TronGrid 的只读 API Key。
7. 保存设置，进入“系统状态”执行连接测试。
8. 测试正常后勾选“启用 USDT-TRC20”。

只填写公开收款地址和只读查询 Key。任何情况下都不要填写私钥、助记词、钱包密码或交易所密码。

## 推荐的系统 Cron

“系统状态”页面会根据站点地址和随机密钥生成完整命令。推荐使用请求头传递密钥：

    * * * * * curl -fsS -X POST -H "X-Jiuliu-Cron-Token: 你的随机密钥" "https://你的域名/wp-json/jiuliu-usdt/v1/cron" >/dev/null 2>&1

插件仍会注册每分钟 WP-Cron 作为兜底。扫描带有 55 秒互斥锁，浏览器轮询也有单支付单限频，不会因为两种任务同时运行而重复结算。

不手工添加服务器 Cron 也能运行：有人访问网站时，WP-Cron 会自动触发；付款弹窗打开时，子比还会轮询支付状态。由于 WP-Cron 不是常驻进程，低访问量网站可能延迟，所以正式收款仍推荐配置上面的系统 Cron。监控接口是本站自己的 `/wp-json/jiuliu-usdt/v1/cron`，它再通过只读 TronGrid API 查询收款地址，不需要监控某一笔交易的独立网址。

## 上线验收顺序

建议依次使用小额真实主网转账验证：

1. 余额充值。
2. 会员购买。
3. 付费文章或下载资源。
4. 商城自动发货商品。
5. 同一交易哈希重复提交应被拒绝。
6. 少付、多付和过期转账应进入人工处理，不能自动发货。
7. 切换支付方式后再向旧金额转账，应进入人工处理。
8. 用户关闭子比订单后转账，应进入人工处理且不得发货；关闭后台的“关闭订单到账观察”开关后，该类订单应停止自动链上查询。

本插件 1.0.0 只连接 TRON 主网，不提供测试网代币模式。因此正式收款地址启用前应将订单金额设得足够小，并使用可承受损失的小额真实 USDT 测试。

## 订单识别与安全

每个待支付订单会生成唯一的六位小数 USDT 金额。自动结算必须同时满足：

- 转入配置的 TRON 地址
- 合约为主网 USDT：TR7NHqjeKQxGTCi8q8ZY4pL8otSzgjLj6t
- TronGrid 返回 only_confirmed 的固化交易
- 代币精度为 6
- 金额与插件支付单完全一致
- 链上时间在订单有效期内
- 交易哈希未被任何其他支付单使用
- 当前子比主支付单和全部关联子订单仍处于待支付状态

链上金额、汇率、交易哈希和核验状态保存在插件独立表中。子比自己的金额表继续只保存人民币两位小数及最终结果。

## 异常处理

- 少付或多付：提交交易哈希后进入人工处理。
- 过期到账：记录交易但不自动发货。
- 用户已切换支付宝/微信等方式：USDT 到账进入人工处理。
- 商城订单已经关闭：禁止自动重新打开，避免库存重复出售。
- 子比结算中断：支付单进入人工处理，管理员先核对权益再决定后续动作。
- 退款：必须由管理员使用外部钱包人工转账。插件没有也不会获得转账权限。

后台“核验/补单”仍会查询 TronGrid，不允许凭空填写一个交易哈希直接发货。即使勾选强制补单，已关闭或状态混乱的商城关联订单仍会被安全拦截。

### 子比订单关闭后的行为

插件监听 Zibll 9.0 原生 `order_closed` 事件，并按稳定的 `payment_id` 同步插件支付单：

- 默认开启“子比订单关闭后继续观察到账”：在过期观察期内继续只读查询；发现到账后绑定交易哈希并转人工处理，不会调用子比发货接口。
- 关闭该开关：停止对已关闭订单的自动链上查询，减少 TronGrid 调用和服务器工作量；管理员仍可在后台粘贴交易哈希核验。
- 如果用户先广播转账、再关闭订单：无论关闭与确认先后如何，结算前都会重新读取主支付单及全部子订单状态；只要已经关闭，就不会自动发货。
- 插件定时任务会调用子比原生超时判断，让主题自己关闭订单、恢复商城库存，再同步插件监控状态。

这里节省的是 API 查询额度和服务器资源，不是 TRON Energy：插件只读链上数据，从不发起链上交易，因此监控本身不消耗 TRX 或 Energy。

## 汇率

- 固定模式：始终使用后台设置的本地货币/USDT 汇率。
- 自动模式：从 CoinGecko 读取 USDT/CNY，验证更新时间并缓存 15 分钟；接口失败、数据过期或偏离备用汇率超过安全阈值时使用固定汇率。
- 汇率加成：正数增加应付 USDT，负数提供折扣。

每张支付单都会锁定创建时的汇率，之后汇率变化不会改变该支付单的精确金额。

CoinGecko 是独立的市场数据聚合服务，并不是 Tether 官方人民币结算价，也不承诺任何交易所最终成交价。插件的自动模式可以一键使用，但正式上线应填写 CoinGecko Demo Key、设置合理备用固定汇率并先点“测试汇率”。希望金额最可控的站点应使用固定模式。

## 其他链

1.0.0 的可执行代码只支持 TRON 主网 USDT-TRC20，其他网络转入不会被识别。架构可以继续增加链适配器：Solana-USDT 通常是下一条最实用的链；Ethereum/Avalanche 等 EVM 网络也能增加，但每条链都必须分别实现官方合约、确认规则、RPC/API 分页、精度、交易解析、地址校验、重放保护和后台设置，不能只改一个网络名称。多链开发属于中等工作量，不需要重写子比结算层。

## 数据保留

卸载插件时不会自动删除支付单、日志和设置，因为这些数据属于财务核对记录。确认不再需要后，可由数据库管理员手动删除：

- {prefix}_jiuliu_usdt_invoices
- {prefix}_jiuliu_usdt_logs
- option：jiuliu_usdt_settings
- option：jiuliu_usdt_db_version

不要在仍存在未处理到账或退款争议时删除这些记录。
